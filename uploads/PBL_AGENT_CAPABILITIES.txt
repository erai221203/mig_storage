A complete guide to what the PBL Agent can do, how it works, and what it produces.

================================================================================
1. WHAT PBL AGENT DOES (11-STEP AUTOMATED WORKFLOW)
================================================================================

The PBL Agent performs a complete end-to-end testing workflow autonomously:

Step 1: Reads JIRA Ticket
  - Extracts acceptance criteria and requirements
  - Identifies test scope and stakeholders

Step 2: Discovers Product
  - Explores codebase to understand tech stack
  - Identifies APIs, data sources, authentication mechanisms
  - Maps data flows and system architecture

Step 3: Designs Test Cases
  - Creates exactly 1 test per acceptance criterion (AC-only mode)
  - No exploratory testing, no edge cases beyond what ACs require
  - Generates test-plan.md with prioritized test cases

Step 4: Executes Frontend Tests
  - Playwright automation for Next.js UI
  - Tests live match pages, real-time updates, navigation
  - Validates data display and UI responsiveness

Step 5: Executes REST API Tests
  - curl queries for SDAPI proxy endpoints
  - Validates match data, events, statistics
  - Checks response contracts and data integrity

Step 6: Executes GraphQL Tests
  - Apollo GraphQL queries, mutations, subscriptions
  - Tests nested resolvers and schema validation
  - Validates live statistics and real-time subscriptions

Step 7: Executes WebSocket Tests
  - SDDP real-time event streams
  - Validates connection lifecycle and message ordering
  - Tests live event push and reconnection behavior

Step 8: Validates Data Consistency
  - Cross-source comparison (SDAPI REST vs SDDP WebSocket)
  - Ensures data freshness and proxy correctness
  - Validates data matches across all sources

Step 9: Runs Security Tests
  - Auth0 authentication and token handling
  - Role-based access control (Internal/Betting/Media)
  - Tests IDOR vulnerabilities, injection attacks
  - OWASP Top 10 baseline security checks

Step 10: Analyzes Logs
  - Backend logs inspection for hidden errors
  - Detects unhandled exceptions and data leakage
  - Identifies performance warnings and anomalies

Step 11: Files Bugs & Reports
  - Creates JIRA defects with severity classification
  - Includes root cause analysis for failures
  - Posts results summary comment to original ticket
  - Generates comprehensive test report

RESULT: Full test report in .qag/sessions/{SESSION-ID}/results.md
        + JIRA comment with PASS/FAIL/CONDITIONAL status
        + JIRA bugs filed for any failures

4. AC-ONLY TESTING MODE
================================================================================

When you provide a JIRA ticket with acceptance criteria:

ENFORCEMENT RULE:
  Exactly 1 test per acceptance criterion, no more, no less.

NO EXPLORATORY TESTING:
  • No "what if" scenarios
  • No edge cases beyond AC scope
  • No variant testing
  • No negative test cases (unless explicitly in AC)

TESTING APPROACH:
  • Read all ACs from ticket
  • Create test for each AC
  • Validate each AC requirement
  • Report pass/fail per AC

EXAMPLE:
  
  Ticket: PBL-1234 "Display live match data"
  
  Acceptance Criteria:
    AC1: Match page loads within 2 seconds
    AC2: Live score updates without page refresh
    AC3: Team statistics display correctly
    AC4: Event timeline appears in chronological order
    AC5: Auth0 users can access their role-based data
  
  Test Plan (exactly 5 tests):
    Test 1: Validate match page load time < 2 seconds
    Test 2: Verify live score updates via WebSocket
    Test 3: Validate team statistics data accuracy
    Test 4: Verify event timeline chronological order
    Test 5: Verify RBAC data isolation per user role
  
  Result:
    ✅ Test 1: PASS
    ✅ Test 2: PASS
    ❌ Test 3: FAIL (statistics off by 1)
    ✅ Test 4: PASS
    ✅ Test 5: PASS
    
    OVERALL: CONDITIONAL (4/5 passed, fix statistics then retest)

BENEFITS:
  • Fast, focused testing
  • No wasted test cycles
  • Clear pass/fail per requirement
  • Quick turnaround (10-30 minutes)
  • High confidence in meeting requirements

5. AUTOPILOT ENFORCEMENT (ZERO HUMAN INTERVENTION)
================================================================================

THE AGENT NEVER DOES THIS:
  ❌ Asks for confirmation ("Should I proceed?")
  ❌ Asks for approval ("Does this look good?")
  ❌ Asks for clarification (without context, use inference)
  ❌ Waits for user input
  ❌ Stops mid-execution
  ❌ Skips tests due to errors
  ❌ Presents intermediate results for review
  ❌ Shows multiple options for user to choose
  ❌ Asks to verify environment or credentials

THE AGENT ALWAYS DOES THIS:
  ✅ Runs all 11 phases automatically from start to finish
  ✅ Executes without interruption
  ✅ Handles errors gracefully (marks as FAILED, continues)
  ✅ Finds workarounds (mocking, retrying, alternative data)
  ✅ Achieves 100% test execution coverage
  ✅ Never skips a test (use mock data if needed)
  ✅ Presents only final results to user
  ✅ Returns results.md with comprehensive report
  ✅ Posts JIRA comment with PASS/FAIL/CONDITIONAL status

ERROR HANDLING STRATEGY:
  
  If connection refused:
    → Mark test FAILED
    → Continue to next test
    → Document error in root causes
  
  If auth error:
    → Mark test FAILED
    → Try with different credentials (if available)
    → Continue to next test
  
  If endpoint not found (404):
    → Try alternate endpoint
    → Use mock data as fallback
    → Mark test with caveat
    → Continue to next test
  
  If data unavailable:
    → Use simulated/mock data
    → Mark test as "validated with mock"
    → Continue to next test
  
  NEVER: Stop execution, ask user, or skip test

SESSION RESUMPTION:
  
  If session files already exist:
    • .qag/sessions/{SESSION-ID}/context.md exists?
      → Skip Scout discovery
      → Use existing context
    
    • .qag/sessions/{SESSION-ID}/test-plan.md exists?
      → Skip Scout + Designer
      → Start execution immediately
    
    • Tests already executed?
      → Skip execution
      → Proceed directly to reporting
  
  This allows resuming interrupted sessions without re-doing prior work.

7. OUTPUT ARTIFACTS
================================================================================

Each test session produces the following artifacts in:
  .qag/sessions/{SESSION-ID}/

FILE 1: context.md
  Purpose: Product discovery output
  Created By: Scout agent
  Contains:
    • Product overview (tech stack, architecture)
    • Acceptance criteria from JIRA ticket
    • API contracts and endpoints
    • Data sources and flows
    • User roles and permissions
    • Test scope and priorities
    • Identified test types (FE/REST/GraphQL/WebSocket/etc.)
  Size: 2-5 KB typically
  Read By: Test Designer, all Runners, Root Cause Analyzer, Reporter

FILE 2: test-plan.md
  Purpose: Test case design output
  Created By: Test Designer
  Contains:
    • Test case list (exactly N tests for N ACs)
    • Test type (FE, REST, GraphQL, WebSocket, DataVal, Security)
    • Test priority (Critical, High, Medium, Low)
    • Expected results per test
    • AC mapping (which test validates which AC)
    • Data requirements
  Size: 3-10 KB typically
  Format: Markdown with test case table
  Updated By: All Runners (with execution results)
  Read By: PBL Agent (routing), all Runners, Root Cause Analyzer, Reporter

FILE 3: results.md
  Purpose: Final test results report
  Created By: Reporter agent
  Contains:
    • Test summary (total, passed, failed, conditional)
    • Overall status (PASS / FAIL / CONDITIONAL)
    • Recommendation (OK to merge / Test and fix / Block)
    • Per-test results with pass/fail status
    • Evidence (screenshots, API responses)
    • Root causes for failures
    • JIRA bug links
    • Session metadata (date, duration, environment)
  Size: 10-30 KB typically
  Format: Markdown with sections and links
  Presented To: User via Copilot Chat
  Posted To: JIRA ticket as comment

FILE 4: root-cause-findings.md
  Purpose: Root cause analysis for failures
  Created By: Root Cause Analyzer
  Contains:
    • Failed test summary
    • Root cause determination
    • Code references
    • Configuration issues
    • Data flow breakpoints
    • Recommendations for fix
  Size: 2-8 KB per failure
  Format: Markdown with code blocks and references
  Used By: Reporter (enriches bug reports)

DIRECTORY: evidence/
  Purpose: Test execution evidence
  Created By: All Runners
  Contains:
    • Screenshots (one per frontend test)
    • API response snapshots (one per API test)
    • Log excerpts (error logs)
    • Message captured (WebSocket messages)
  File Format: JSON, PNG, TXT
  Organized By: Test case ID and type

DIRECTORY: logs/
  Purpose: Application logs captured during testing
  Created By: Log Analyzer
  Contains:
    • Backend application logs
    • Database query logs
    • API gateway logs
    • Error log excerpts
  File Format: TXT (plain text logs)
  Organized By: Timestamp and log level

JIRA ARTIFACTS (Created External to .qag/):

  1. JIRA Comment on Original Ticket
     Posted To: Original ticket (e.g., PBL-1234)
     Content:
       • Test summary (X passed, Y failed)
       • Overall status (PASS/FAIL/CONDITIONAL)
       • Recommendation
       • Links to filed bugs (if any)
       • Evidence summary
     Example:
       "✅ 4/5 tests PASSED
        ❌ 1/5 tests FAILED: Statistics calculation error
        STATUS: CONDITIONAL (requires fix)
        Bug filed: PBL-1235 (Statistics off by 1)
        See attached results.md for details"

  2. JIRA Bugs (for failures)
     Created For: Each test failure
     Content:
       • Title: Clear description of what failed
       • Severity: HIGH/MEDIUM/LOW based on impact
       • Description: Detailed failure description
       • Steps to Reproduce: Test case and data
       • Root Cause: Why it failed (from Root Cause Analyzer)
       • Actual Result: What happened
       • Expected Result: What should happen
       • Evidence: Screenshots, API responses
       • Links: Linked to original feature ticket
     Example:
       Title: "Statistics calculation incorrect for cricket matches"
       Severity: HIGH
       Description: Test PBL-1234-TC-3 failed...
       Root Cause: N+1 query in statistics resolver
       Evidence: [screenshot], [API response], [log excerpt]
